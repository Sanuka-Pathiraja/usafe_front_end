Backend Updates Needed For Frontend Step-Accurate Status
=========================================================

Goal
----
Frontend now expects each important emergency task to report real success/failure.
The UI should not show a step as completed unless backend confirms success.

Affected frontend flow:
1. Message all emergency contacts
2. Call contacts sequentially (1..N)
3. Wait gap
4. Call 119


Required Backend Behavior
-------------------------

1) POST /emergency/start
- Must create session and attempt messaging contacts.
- Must return session id always when session creation succeeds.
- Must explicitly report messaging result:
  - success true/false (or ok true/false)
  - message string for operator-readable reason
  - optional machine code

Recommended response:
{
  "sessionId": "abc123",
  "success": false,
  "message": "Insufficient SMS balance",
  "code": "LOW_FUNDS",
  "messaging": {
    "attempted": 5,
    "sent": 0,
    "failed": 5
  }
}

Important:
- If low funds (or similar), return success=false with clear message/code.
- Do not return ambiguous success if SMS was not actually sent.


2) POST /emergency/:sessionId/call/:contactIndex/attempt
- Must return whether that specific call was answered.
- Existing contract is good:
{
  "answered": true
}
or
{
  "answered": false
}

Recommended optional fields:
{
  "answered": false,
  "status": "no-answer|busy|failed|timeout",
  "message": "Carrier timeout"
}

Important:
- 404 for missing contact index/session is acceptable.
- Frontend treats 404 as non-fatal for a contact call and proceeds to next contact.


3) POST /emergency/:sessionId/call-119
- Must explicitly report whether 119 was actually called.
- Frontend now checks:
  - ok/success != false
  - emergencyServicesCalled != false

Recommended response:
{
  "ok": true,
  "success": true,
  "emergencyServicesCalled": true,
  "message": "119 call placed"
}

Failure example:
{
  "ok": false,
  "success": false,
  "emergencyServicesCalled": false,
  "code": "CARRIER_BLOCK",
  "message": "Provider rejected destination"
}


Error Message Quality
---------------------
Frontend now surfaces error text directly in failed step rows.
Use short clear messages, e.g.:
- "Insufficient SMS balance"
- "Session expired"
- "Call provider timeout"
- "119 route unavailable"


State Tracking (recommended)
----------------------------
Persist per session:
- sessionId
- userId
- messaging.status + reason
- callAttempts[index].status + reason
- someoneAnswered
- answeredByContactIndex
- emergencyServicesCalled
- timestamps

Prefer DB/Redis over in-memory map for production reliability.


Compatibility Notes
-------------------
Frontend callback expectations now:
- onMessageAllContacts -> returns success + optional message
- onCallContact -> returns answered boolean
- onCall119 -> returns success + optional message

If backend only returns HTTP 200 with no status fields, frontend may still
assume success. Add explicit status fields to avoid false positives.
