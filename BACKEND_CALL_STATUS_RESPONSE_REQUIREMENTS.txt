Backend Response Requirements For Accurate Call Status UI
=========================================================

Why this is needed
------------------
Frontend now differentiates:
- normal call outcome (no answer / busy / timeout) vs
- true provider/system errors (should show FAILED on step)

If backend sends only `answered: false`, frontend cannot know if it was
an expected no-answer or a provider failure.


Endpoint
--------
POST /emergency/:sessionId/call/:contactIndex/attempt


Required response fields (recommended on every 2xx response)
------------------------------------------------------------
{
  "success": true,
  "answered": false,
  "finalStatus": "no-answer",
  "message": "No answer within timeout",
  "code": "NO_ANSWER"
}

Field meanings:
- success:
  true  -> call attempt executed correctly (even if not answered)
  false -> call attempt failed due to provider/system error

- answered:
  true when contact answered
  false when not answered or failed

- finalStatus (normalized):
  answered | no-answer | busy | timeout | failed | not-configured

- message:
  short human-readable reason for UI

- code:
  machine-readable code for analytics/debugging


Status mapping guidelines
-------------------------
1) Contact answered
{
  "success": true,
  "answered": true,
  "finalStatus": "answered",
  "message": "Call answered",
  "code": "ANSWERED"
}

2) No answer / busy / timeout (not system error)
{
  "success": true,
  "answered": false,
  "finalStatus": "no-answer",
  "message": "No answer within timeout",
  "code": "NO_ANSWER"
}

{
  "success": true,
  "answered": false,
  "finalStatus": "busy",
  "message": "Line is busy",
  "code": "BUSY"
}

{
  "success": true,
  "answered": false,
  "finalStatus": "timeout",
  "message": "Call timed out",
  "code": "TIMEOUT"
}

3) Provider/system error (this should display FAILED in UI step)
{
  "success": false,
  "answered": false,
  "finalStatus": "failed",
  "message": "Provider rejected request",
  "code": "CALL_PROVIDER_ERROR"
}


HTTP status usage
-----------------
- 2xx: request processed, include full status payload above
- 404: session/contact not found (frontend currently treats missing contact as non-fatal)
- 5xx: unexpected server failure

Important:
- Do not return 2xx with ambiguous payload when provider error happened.
- Include `success:false` + clear `code/message` for provider errors.


Frontend behavior this enables
------------------------------
- `success:true + answered:false` -> step can show normal non-answered progression
- `success:false` -> step shows FAILED with backend message, but flow continues to next contact
- `answered:true` -> remaining contact calls are skipped


Minimum backend implementation checklist
----------------------------------------
1) Add `success` and `finalStatus` to call attempt responses.
2) Emit `code/message` consistently for failures.
3) Normalize provider statuses to:
   answered, no-answer, busy, timeout, failed, not-configured.
4) Keep response shape stable across all call providers.
