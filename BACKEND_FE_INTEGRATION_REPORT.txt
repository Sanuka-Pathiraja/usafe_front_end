Current Observed Production Issue (2026-02-19)
- Recipients are receiving only: "USafe emergency alert"
- Location/time lines are missing.
- Root cause found: FE had been sending a fixed `message` field, so backend correctly treated it as full override and skipped default template composition.
- FE fix applied: FE no longer sends `message` by default.

Emergency SMS Message (backend default)
If message is not provided, backend sends:
USafe Alert ⚠️
Danger detected
Location: <locationText or Unknown location>
Time: <dangerTime/time/eventTime or current server time>

Backend Change Request (to prevent future regressions)
1) Keep current behavior: if `message` is absent, always build full template with location/time.
2) If `message` is present, support one of these safe options:
   - Option A (recommended): still append normalized location/time lines unless `messageMode=exact` is explicitly set.
   - Option B: add `useDefaultTemplate=true|false` flag and ignore raw `message` unless `useDefaultTemplate=false`.
3) Return a response field confirming rendered content strategy, for example:
   - `renderMode: "default-template" | "custom-message"`
4) Ensure logs include whether location/time were included in final outbound SMS.

Error Handling Contract
Non-2xx errors are consistent JSON:
{
  "ok": false,
  "success": false,
  "code": "<ERROR_CODE>",
  "message": "<human readable>"
}

UTF-8 / Encoding
Backend JSON responses are explicitly UTF-8:
Content-Type: application/json; charset=utf-8

Emoji handling note:
- Use valid UTF-8 emoji only.
- Do not emit "??" placeholders.
- If emoji cannot be guaranteed, send plain-text fallback.
- FE now sends `unicode: true` for emergency start to ensure emoji messages render correctly.

FE Notes
If /emergency/start is non-2xx, sessionId will not exist, so FE must stop flow and display backend message.
FE should treat PARTIAL_SEND as session-created but with warning state.
On UNAUTHORIZED, FE should logout and redirect to login.
